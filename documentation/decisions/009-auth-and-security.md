# ADR-009 — Auth & security (2026-07-10)

## 009.1 RS256 + JWKS local verification (v4); introspection becomes opt-in
**Context:** per-request HTTP introspection made auth-service a whole-API SPOF
— proven live twice (breaker outage incidents; review §2). **Decision:**
auth-service signs RS256 with `kid` headers and serves /.well-known/jwks.json;
api-service verifies locally with a cached JWKS; the introspection endpoint
remains for revocation-strict routes, opt-in. Ships in v4. **Consequences:**
the hop, the breaker class and the validate-rate-limit class disappear;
key rotation becomes a drill; core of the v8 auth template.

## 009.2 Revocation: short access TTL + rotating refresh sessions
**Decision:** access tokens ≤15m; refresh tokens are one-time-use, rotated
against a DB sessions table (reuse detection = theft signal); logout deletes
the session. No per-request denylist — 15m revocation lag accepted.
**Consequences:** logout becomes real; stateless hot path preserved.

## 009.3 Secrets: layered — generate-at-setup → SOPS → Secrets Manager
**Decision:** local secrets generated by `make init-secrets` into .env / k8s
Secrets (never committed); anything that must live in-repo is SOPS+age
encrypted; AWS uses Secrets Manager fed by Terraform, synced into the cluster
via external-secrets. **Consequences:** the committed-secrets habit (review
§11) ends structurally; one pattern from laptop to cloud (matches ADR-007.6).

## 009.4 TLS: cert-manager at kind ingress; no service mesh
**Decision:** v2/v3 ingress serves TLS from a local CA via cert-manager (real
certificate workflows, renewal drills); pod-to-pod stays plaintext; mesh/mTLS
explicitly out of scope. AWS edge TLS is ACM (ADR-006.6). **Consequences:**
cert practice without operating a mesh on a laptop.

## 009.5 Rate limiting moves to the ingress layer
**Context:** express-rate-limit counters are per-instance memory — wrong at
>1 replica; with JWKS (009.1) the hot validate path is gone anyway.
**Decision (owner call):** rate limiting enforced via nginx-ingress
annotations (per-IP) on auth endpoints; the app keeps only account-lockout
logic. Compose mode retains the existing in-app limiter as-is (no ingress
there). **Consequences:** infra-layer limiting practice; app simplifies;
lockout counters remain in-app (move to Redis only if auth-service scales
past one replica in drills — revisit then).

## 009.6 Drop the users.salt column
**Context:** bcrypt already salts; a second same-row salt adds nothing
(review §7). **Decision:** remove the column and the concat logic in the v4
auth work; no pepper. **Consequences:** simpler schema and verification path.

## 009.7 Roles: minimal real enforcement
**Decision:** role middleware + a small admin-only surface (user management,
destructive endpoints); seed one admin user. No RBAC system. **Consequences:**
the JWT role claim stops being decorative; template ships enforcement
middleware.
